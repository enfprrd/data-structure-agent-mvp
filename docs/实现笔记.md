## 1. 项目总体目标

本项目目标是构建一个面向本科《数据结构》课程的智能助教原型系统。系统由三部分协同工作：

- 本地 Markdown 知识库：提供教材风格的可控知识来源。
- DeepSeek Chat：负责自然语言问答、代码生成、含糊追问理解和演示意图解析。
- 本地可视化模拟器：根据结构化演示请求，生成可逐步播放的数据结构状态变化。

项目不是单纯聊天机器人，而是“RAG 问答 + C 代码教学 + 数据结构过程演示”的组合系统。

## 2. 页面主流程

主入口是 `app.py`。

页面由三块组成：

- 左侧 sidebar：只保留运行状态、账号、上下文状态、恢复生成状态和清空按钮。
- 主页面“问答演示”页签：左侧显示用户和助教对话，右侧显示当前可视化演示、步骤切换、自动播放和 OperationRequest JSON。
- 主页面“知识库阅读”页签：提供章节选择、小节阅读和关键词搜索。

用户输入问题后，主流程大致如下：

1. 接收用户问题。
2. 判断是否为含糊追问。
3. 必要时调用 DeepSeek 将追问改写成独立问题。
4. 让 DeepSeek 为当前问题输出本轮 RAG 检索计划。
5. 用改写后的问题、RAG 检索计划和低权重历史关键词执行加权 RAG 检索。
6. 将检索片段、当前问题和历史消息发送给 DeepSeek。
7. DeepSeek 生成教学回答和必要的 C 代码。
8. 本地检查 C 代码是否存在、是否能编译运行。
9. DeepSeek 的回答和 C 代码再交给演示意图解析器。
10. 解析器输出 OperationRequest。
11. dispatcher 按 structure 和 operation 分发到具体模拟器。
12. 模拟器输出 VisualizationTrace。
13. HTML renderer 将当前 step 渲染到右侧演示板。

这个流程的核心是把“自然语言问题”逐步压缩成“结构化、可验证、可播放”的本地演示轨迹。

## 3. RAG 检索实现

RAG 模块位于 `rag.py`，核心类是 `MarkdownKeywordRetriever`。

### 3.1 知识库组织

知识库位于 `knowledge/`，每个 Markdown 文件对应一个课程章节或主题：

- `introduction.md`
- `linear_list.md`
- `sequential_list.md`
- `linked_list.md`
- `stack_queue.md`
- `string.md`
- `array_generalized_list.md`
- `tree_binary_tree.md`
- `graph.md`
- `search.md`
- `internal_sorting.md`
- `external_sorting.md`

每个文件按 Markdown 标题切分。一级标题代表章节，二级标题代表小节。

### 3.2 切片策略

最初只按 `##` 切分小节。后来发现小节脱离章节标题后会丢失主题，例如“插入操作”不知道属于顺序表还是链表。

当前策略：

- 每个 `##` 小节前自动补上所属 `#` 一级标题。
- 检索结果中保留 `source`，格式类似 `internal_sorting.md#片段9`。
- 小节内容用于模型回答，也用于用户在 sidebar 中检查本次检索结果。

这样可以减少“同名操作”在不同章节之间的混淆。

### 3.3 关键词和评分

检索使用关键词评分，不依赖向量数据库。原因是项目规模较小，Markdown 笔记结构明确，关键词检索可解释性强。

评分因素包括：

- 查询 token 与片段 token 的重合。
- 领域词在 query 和 chunk 中同时出现。
- 领域词在标题中出现时额外加权。
- 如果命中的领域词也出现在小节首标题中，再加一层权重。
- 上一轮 RAG 关键词命中片段或标题时额外加权。

为了降低噪声，加入中文虚词过滤，例如：

- `的`
- `了`
- `是`
- `什么`
- `怎么`
- `吗`

### 3.4 短语归一和别名

为了处理中文问法差异，RAG 做了轻量归一：

- 去除短语中的空格：`B树` 可匹配 `B 树`。
- 将 `和` 与 `与` 归一：`B树和B+树` 可匹配 `B 树与 B+ 树`。
- 设置别名：
  - `归并算法 -> 归并排序`
  - `合并排序 -> 归并排序`

### 3.5 历史上下文污染问题

曾经的检索输入是：

```text
当前问题 + 此前完整对话上下文
```

这导致用户前面聊了很多链表后，再问“栈是什么”，RAG 会被历史中的链表关键词带偏。

当前改为：

- RAG 默认只检索当前有效问题。
- 历史对话仍发送给 DeepSeek 作为聊天上下文。
- 对“演示一下呗”这类当前问题没有主题的追问，先改写成独立问题，再检索。
- 检索前先让 DeepSeek 为当前问题输出 RAG 检索计划。
- 历史 RAG 关键词不能直接压过当前问题，只有 DeepSeek 判断仍相关时才低权重参与。

### 3.6 DeepSeek RAG 检索计划

为了避免“如果是更多的结点呢”这类短追问被单个词带偏，也避免上一轮主题污染新问题，当前采用“DeepSeek 检索规划 + 加权关键词检索”的策略。

检索前，系统调用 DeepSeek 输出本轮 RAG plan。格式类似：

```json
{
  "query_keywords": ["二叉树", "后序遍历", "遍历"],
  "groups": [
    {
      "name": "tree_traversal",
      "keywords": ["树", "二叉树", "后序遍历"],
      "weight": 6
    }
  ],
  "history_keywords": []
}
```

字段含义：

- `query_keywords`：当前问题最重要的检索关键词，权重最高。
- `groups`：DeepSeek 给出的 RAG 关键词组，每组有 `weight`，范围为 1 到 6。
- `history_keywords`：DeepSeek 判断仍和当前问题相关的历史关键词；如果当前问题是新主题，应为空。

检索时执行三类查询：

- 当前有效问题检索，并用 `query_keywords` 高权重增强。
- RAG plan 全部关键词检索。
- 当前有效问题 + RAG plan 关键词组合检索。

`rag.py` 的 `MarkdownKeywordRetriever.retrieve()` 支持 `boost_weights`。权重进入片段评分：

- 当前问题关键词通常权重为 6。
- DeepSeek 分组关键词使用组内 `weight`。
- 历史关键词最多低权重参与，不能覆盖当前主题。

重要约束：

- 不再使用本地 `RAG_TOPIC_GROUPS` 或本地主题过滤表判断“这是不是树/图/顺序表”。
- 语义判断交给 DeepSeek；本地只负责按 RAG plan 做可解释的加权检索。
- 如果 DeepSeek 不可用，系统退回只按当前有效问题检索，避免用历史关键词强行猜主题。

每轮回答后仍会让 DeepSeek 总结隐藏 RAG 关键词并保存到消息中，供下一轮 RAG plan 判断是否继承。

### 3.7 知识库阅读

用户不只通过问答间接接触 `knowledge/`，也可以在“知识库阅读”页签中直接阅读知识库资料。

当前阅读功能包括：

- 选择知识库章节。
- 按 Markdown 小节展开阅读。
- 输入关键词跨章节搜索相关小节。
- 保留本次 RAG 检索片段展示，方便对照“系统这次引用了哪些资料”。

这样用户可以自行翻阅、复查和学习资料，而不是只能被动等待模型回答。

阅读入口后来从 sidebar 移到主页面独立页签，避免资料阅读挤占侧边栏，也让问答演示和自主阅读的工作流更清晰。

## 4. 含糊追问改写

含糊追问处理在 `app.py` 中。

典型问题：

```text
用户：归并算法的时间空间复杂度是多少
用户：演示一下呗
```

如果只用第二句检索，里面没有“归并排序”，会命中不相关内容。

当前策略：

1. 本地只做一个轻量判断：这句话是否可能是含糊追问。
2. 如果是，就调用 DeepSeek 根据最近对话改写成独立问题。
3. 改写结果用于 RAG 检索和用户 prompt。
4. 如果 DeepSeek 不可用，就使用用户原问题，不再本地拼接上一轮主题。

改写 prompt 的约束：

- 只输出改写后的问题。
- 必须继承最近一轮具体主题。
- 保留算法或数据结构名称。
- 不要擅自改成链表、顺序表等无关主题。
- 无法判断时原样输出用户新问题。

这样避免用大量硬编码正则去猜主题，也更适合后续扩展。

## 5. DeepSeek 回答生成

DeepSeek 调用封装在 `llm.py` 的 `DeepSeekClient`。

支持：

- 从 `.env` 读取 `DEEPSEEK_API_KEY`。
- 从 Streamlit secrets 读取 `DEEPSEEK_API_KEY`。
- 调用 `https://api.deepseek.com/chat/completions`。
- 设置模型为 `deepseek-chat`。

系统提示词位于 `prompts/system_prompt.txt`，主要约束模型：

- 作为本科数据结构课程助教回答。
- 默认中文。
- 默认 C 语言代码。
- 严格依据本地知识库片段。
- 资料不足时说明知识库不足。

`build_user_prompt()` 会将检索片段和用户问题组合成最终用户 prompt。

对于演示类问题，prompt 额外要求模型回答包括：

- 初始状态。
- 操作目标。
- 分步过程。
- 关键指针或数组移动原因。
- 最终结果。
- 易错点。
- 完整可运行 C 程序。

## 6. C 代码检查

C 代码检查位于 `code_checker.py`。

设计目的：

- 保证演示类回答真的包含 C 代码。
- 将 C 代码作为演示意图解析的重要依据。
- 减少 DeepSeek 文字和代码不一致的问题。

流程：

1. 从回答中提取 ```c 代码块。
2. 保存到临时文件。
3. 调用本地编译器检查。
4. 如果编译或运行失败，将错误反馈给 DeepSeek 重新生成。

如果用户只是普通问答，没有明确演示意图，则代码检查要求会降低。

## 7. DSVP 协议设计

可视化协议位于 `visualizer/protocol.py`。

DSVP 可以理解为 Data Structure Visualization Protocol，是本项目内部使用的结构化演示协议。

### 7.1 OperationRequest

OperationRequest 表示一次演示请求，主要字段：

- `version`
- `structure`
- `operation`
- `params`
- `initial_state`
- `options`

示例：

```json
{
  "version": "1.0",
  "structure": "sort",
  "operation": "merge_sort",
  "params": {
    "values": [5, 2, 4, 1]
  },
  "initial_state": {
    "data": [5, 2, 4, 1],
    "metadata": {}
  },
  "options": {
    "language": "c",
    "explain_level": "beginner"
  }
}
```

### 7.2 VisualizationTrace

VisualizationTrace 表示模拟器输出的完整演示轨迹：

- `title`
- `structure`
- `operation`
- `summary`
- `steps`
- `errors`
- `warnings`

每个 step 包含：

- `step_id`
- `phase`
- `title`
- `description`
- `state`
- `highlights`
- `actions`
- `code_refs`
- `message`

### 7.3 校验策略

协议使用 Pydantic 校验。

设计上区分两类情况：

- JSON 格式不合法或缺少必要参数：反馈给 DeepSeek 重新生成。
- JSON 合法但本地模拟器不支持：前台提示“当前操作暂不支持可视化演示”。

这样可以避免把“本地没实现”和“模型输出错了”混在一起。

## 8. 演示意图解析

演示意图解析器位于 `visualizer/intent_parser.py`。

输入是：

- 此前完整对话上下文。
- 用户新问题。
- 助手刚刚输出的教学回答。
- 教学回答末尾的 C 代码。

输出是：

- `None`：没有演示意图。
- `Clarification`：有演示意图但参数不完整。
- `OperationRequest`：完整合法演示请求。

解析 prompt 的关键规则：

- 只输出 JSON。
- 优先依据 C 代码中的 main、初始化数据和函数调用。
- 文字和代码冲突时，以 C 代码为准。
- 如果用户要演示但本地尚不支持，也要输出合法 OperationRequest。
- 图演示的顶点标签必须由 DeepSeek 在 OperationRequest 中显式保留；用户给出数字顶点时，必须写入 `metadata.vertices`，本地不再从原文二次猜测或替换标签。
- 图使用邻接矩阵时，必须同时给出矩阵行列对应的 `metadata.vertices`；DFS/BFS 起点也必须和顶点标签一致。
- 无权图不要凭空生成 `weight`；邻接表形式如 `1: 2,3` 时，逗号后的数字表示邻接点，不表示边权。

本地支持列表通过 `supported_demo_summary()` 动态注入 prompt，让 DeepSeek 知道哪些组合能播放。

## 9. dispatcher 分发

分发器位于 `visualizer/dispatcher.py`。

逻辑很简单：

1. 检查 `structure + operation` 是否为本地支持组合。
2. 不支持则返回 unsupported trace。
3. 支持则按 structure 找对应模拟器。

当前 structure 包括：

- `sequential_list`
- `singly_linked_list`
- `stack`
- `queue`
- `binary_tree`
- `graph`
- `search_table`
- `sort`

## 10. 本地模拟器实现

模拟器位于 `visualizer/simulators/`。

每个模拟器都是纯 Python 逻辑，不调用 DeepSeek。这样可以保证右侧动画可重复、可验证。

### 10.1 顺序表

文件：`visualizer/simulators/sequential_list.py`

支持：

- 插入。
- 删除。
- 查找。

特点：

- 用数组 cells 表示顺序表。
- 高亮当前比较、移动、插入或删除位置。
- 插入和删除会逐步展示元素移动。

### 10.2 单链表

文件：`visualizer/simulators/singly_linked_list.py`

支持：

- 插入。
- 删除。
- 查找。
- 头插法建表。
- 尾插法建表。

特点：

- 默认带头结点。
- state 中包含 nodes 和 pointers。
- 渲染器根据指针展示 head、pre、p、s 等位置。
- 插入时强调 `s->next = pre->next` 必须先于 `pre->next = s`。

### 10.3 栈基础操作

文件：`visualizer/simulators/stack.py`

支持：

- 顺序栈 push。
- 顺序栈 pop。

特点：

- state.kind 为 `stack`。
- top 用下标表示。
- push 检查栈满，pop 检查栈空。

### 10.4 栈经典应用

文件：`visualizer/simulators/stack_applications.py`

该文件用于承载大量栈应用，避免把所有应用硬塞进基础 push/pop。

支持：

- 链栈入栈、链栈出栈。
- 进制转换。
- 括号匹配。
- 表达式括号合法性检查。
- 中缀转后缀。
- 后缀表达式求值，也就是逆波兰表达式。
- 前缀表达式求值。
- 运算符优先级表达式求值。
- 函数调用栈。
- 递归调用过程。
- 汉诺塔递归。
- 汉诺塔非递归栈模拟。
- 迷宫回溯。
- 非递归 DFS。
- 二叉树非递归先序、中序、后序遍历。
- 浏览器前进后退。
- 撤销重做。
- HTML/XML 标签匹配。
- 语法分析栈。
- 单调栈。
- 下一个更大元素。
- 柱状图最大矩形。
- 每日温度。
- 栈排序。
- 两个栈实现队列。
- 栈逆序。
- 火车调度。
- 出栈序列合法性判断。

这些演示统一使用 `state.kind = stack_app`。

stack_app state 包括：

- `stacks`：一个或多个栈。
- `input`：输入序列。
- `output`：输出序列。
- `current`：当前处理对象。
- `note`：补充说明。

### 10.5 队列

文件：`visualizer/simulators/queue.py`

支持：

- 入队 enqueue。
- 出队 dequeue。

特点：

- 显示 front 和 rear。
- 入队高亮队尾。
- 出队高亮队头。

### 10.6 二叉树

文件：`visualizer/simulators/binary_tree.py`

支持：

- 先序遍历。
- 中序遍历。
- 后序遍历。
- 层序遍历。

输入采用层序数组：

```text
["A", "B", "C", null, "D"]
```

当前 state 同时包含：

- `nodes`：真实非空节点。
- `slots`：完整层序槽位，包括空节点。
- `edges`：父子边。

增加 slots 的原因是二叉树绘制时需要识别空节点位置，否则树形结构会压缩，造成父子连线和教材习惯不一致。

### 10.7 图

文件：`visualizer/simulators/graph.py`

支持：

- 建图。
- DFS。
- BFS。
- Dijkstra。
- Prim 最小生成树。
- Kruskal 最小生成树。
- Floyd 多源最短路径。
- 拓扑排序。

输入可以是：

- 邻接矩阵。
- 边列表。
- 字典形式边。

state 包含：

- `nodes`
- `edges`
- `visit_order`

图的邻接表由 `_parse_graph()` 构造。

Prim 演示从一个起点开始维护已选顶点集合，每一步选择连接集合内外顶点的最小边。Kruskal 演示先按边权排序，再用并查集思想判断是否成环。Floyd 演示逐步允许更多中间顶点更新距离矩阵。拓扑排序演示入度表、入度为 0 的顶点输出和删除出边过程。

### 10.8 查找表

文件：`visualizer/simulators/search_table.py`

支持：

- 顺序查找。
- 折半查找。

特点：

- 顺序查找逐个比较。
- 折半查找展示 low、mid、high 的变化。

### 10.9 排序

文件：`visualizer/simulators/sort.py`

支持：

- 冒泡排序。
- 直接插入排序。
- 简单选择排序。
- 快速排序。
- 归并排序。

归并排序实现重点：

- 递归二分区间。
- 单元素区间作为天然有序。
- 合并两个有序段。
- 用辅助数组 `aux` 暂存结果。
- 将辅助数组写回原数组。
- 时间复杂度为 `O(n log n)`。
- 空间复杂度为 `O(n)`。

排序演示会同步展示 C 代码中的下标变量和游标：

- 冒泡排序显示 `i`、`i+1`、`end`。
- 插入排序显示 `i`、`j`、`insert`。
- 选择排序显示 `i`、`j`、`min`。
- 快速排序显示 `low`、`high`、`i`、`j`、`pivot`。
- 归并排序显示 `left`、`mid`、`right`、`i`、`j`、`k`。

这些游标放在 `state.pointers` 中，HTML renderer 会在数组格子上方显示标签，并在指针面板列出变量当前指向的位置。

## 12. 响应式布局

响应式布局主要在 `app.py` 和 `html_renderer.py` 中实现。

### 12.1 桌面和 HD 屏

问题：

- HD 屏幕高度不够时，长演示会撑出页面。
- 右侧演示板如果没有独立滚动，用户无法查看后续内容。

解决：

- 主页面使用 `100dvh`。
- 问答页采用左右双栏：左侧聊天问答区，右侧演示板。
- 聊天区和演示区分别独立滚动。
- 聊天输入框固定在左栏底部，消息列表只在中间区域滚动。
- `检索片段` 放入消息滚动区内，避免把输入框顶出视口。
- 左栏容器需要撑满 Streamlit column 的父层；必要时使用 `:has(> .st-key-chat_shell)` 将 Streamlit 包裹层高度补齐。
- 小高度桌面屏降低标题字号和间距。
- 演示板设置固定高度和内部滚动。
- 历史消息中的演示步骤不能只保存 step index。新消息需要保存 `demo_request`，点击历史步骤时先恢复 OperationRequest，再 dispatch 出右侧演示轨迹。
- 旧历史消息没有 `demo_request` 时，如果右侧没有当前演示数据，应隐藏同步步骤入口，避免左侧有步骤而右侧仍是等待态。

### 12.2 手机

手机端策略：

- 页面允许纵向滚动。
- 聊天区高度限制为视口比例。
- 演示板高度限制为视口比例。
- 演示板内部独立滚动。
- 演示元素缩小，减少横向溢出。

## 16. 后续可写入论文的技术点

### 16.1 可控知识源

系统不直接依赖模型自由发挥，而是通过本地 Markdown 教材笔记约束回答范围。这可以作为“面向课程场景的可控 RAG 问答”进行描述。

### 16.2 自然语言到可视化协议

DeepSeek 不直接生成动画，而是生成 OperationRequest。动画由本地模拟器执行。这种设计把“语义理解”和“状态模拟”解耦，提高了可验证性。

### 16.3 本地模拟器保证正确性

数组移动、链表指针、栈顶变化、图遍历等由 Python 逻辑生成，而不是由大模型逐步编造。因此演示过程更稳定。

### 16.4 代码与演示对齐

演示类回答要求生成完整 C 程序，解析器优先读取 C 代码中的 main 和实参。这可以减少文字讲解和动画参数不一致的问题。

### 16.5 含糊追问改写

系统使用 DeepSeek 对“演示一下呗”等上下文依赖问题进行改写，改写后再检索知识库。这是多轮教学对话中的重要机制。

### 16.6 结构化可视化轨迹

VisualizationTrace 将每一步状态、动作、高亮、代码线索统一起来，便于前端渲染，也便于测试。

### 16.7 轻量部署

系统只依赖 Streamlit、requests、pydantic 和本地 Markdown 文件，可以较轻量地部署到 Streamlit Cloud。

## 17. 当前局限

- RAG 仍是关键词检索，没有使用向量检索或混合检索。
- DeepSeek 改写追问需要额外一次模型调用，会增加延迟。
- 图布局是圆形布局，复杂图可能仍会交叉。
- 二叉树使用层序数组输入，对任意指针式树结构还需要进一步适配。
- C 代码检查依赖部署环境中的 gcc。
- 演示器覆盖了很多典型算法，但仍未覆盖所有教材内容，例如外部排序完整动画、B 树插入删除、AVL 旋转等。

## 19. PPT-Aware Learning 第一阶段 MVP

### 19.1 模块边界

`ppt_learning.py` 负责 PPT 学习模式的核心逻辑：

- `SlideCard`：一页 PPT 的结构化卡片，字段包括 `deck_id`、`slide_id`、`title`、`raw_text`、`notes`、`tables`、`code_blocks`、`summary`、`keywords`、`concept_type`。
- `ContextPack`：当前答疑轮次发送给模型的上下文包，包含当前页完整卡片、前后页摘要、目录、相关页、会话摘要和少量教材片段。
- `parse_pptx_to_slide_cards()`：使用 `python-pptx` 本地解析 PPTX。
- `build_context_pack()`：按当前页和用户问题构造 page-aware RAG 上下文。
- `answer_with_context_pack()`：把 ContextPack 转成严格文本版 Prompt 调用 DeepSeek。

### 19.2 解析策略

第一阶段只做文本解析：

- 提取 slide title、文本框、表格和 notes。
- `slide_id` 使用 1-based 页码，便于 UI 和引用来源页码一致。
- `tables` 使用三层列表保存：多个表格 -> 行 -> 单元格文本。
- `code_blocks` 先识别 Markdown fenced code 和缩进代码，后续可扩展为更强的代码片段识别。
- 如果页面没有正文、备注和表格，`concept_type` 标记为 `text_missing`，summary 明确说明不猜测图片信息。

### 19.3 DeepSeek 调用约束

DeepSeek API 只按文本调用：

- 导入 PPT 后，元数据生成以“单页文本”为单位调用，不把整份 PPT 一次性发送给模型。
- 当前页答疑只发送 ContextPack：当前页完整文本、前后页摘要、标题目录、检索到的相关页和会话摘要。
- Prompt 明确禁止声称看过截图、图片或视觉布局。
- 若 PPT 文本中没有相关信息，回答必须说明“不在当前 PPT 文本中”。
- 回答必须标注来源页码。

### 19.4 Streamlit 页面

`app.py` 新增“PPT学习模式”页签：

- 支持上传 PPTX，也支持输入本地 PPTX 路径。
- 左侧显示当前页标题、页码、提取文本、notes、表格和 SlideCard JSON。
- 支持上一页/下一页。
- 右侧保留 PPT 独立对话，不混用普通问答历史。
- “讲解当前页”按钮默认围绕当前页 ContextPack 生成讲解。

### 19.5 后续扩展点

- OCR：可在 `parse_pptx_to_slide_cards()` 之后为 `text_missing` 页补充 OCR 文本，再重新生成 summary/keywords。
- 视觉模型：可增加独立 `visual_summary` 字段，但需要模型支持图片输入；当前 DeepSeek 文本接口下不启用。
- 检索增强：`retrieve_related_slides()` 当前是轻量关键词检索，后续可替换为 BM25/向量/混合检索，`ContextPack` 接口无需大改。
- 持久化：当前 SlideCard 存在 session 中，后续可写入 SQLite 或本地 JSON 缓存，避免重复导入时重新调用元数据生成。
